# Running Gates

Gates are not a formality. They are the proof that the spec's requirements were met, generated by someone other than the agent that made the claim. The gate report is most trustworthy when CI-attested; an author-generated report is the fallback, not the goal.

<HARD-GATE>
Do NOT create a PR or declare work complete until a gate report exists with verdicts for all Verification items. An agent claiming its own work is done is not a completion signal.
</HARD-GATE>

## TRIGGER

Invoke this skill when:

- the user says "done?", "ship it", "ready to merge?", "open the PR", "is this done"
- you believe a plan is complete and want to verify before claiming so
- "run gates", "run the gates", "gate check"

Do NOT trigger if there is no spec with a Verification section — there is nothing to run against.

---

## Pre-flight checks

Before executing any gate:

**1. Check for open escalations.**

Read `specs/<NN>-<slug>/escalations/`. Any file with `status: open` blocks the gate run. Report them:

> Gate run blocked: N open escalation(s): [list titles and ids]. Resolve before running gates.

Do not proceed until all escalations are either `resolved` or explicitly scoped to items not covered by this gate run.

**2. Read the spec's Verification section completely.**

The spec is the source of truth. Read every verification item before executing any. You are executing this list, not your memory of it.

**3. Record the completion claim first.**

Before executing a single gate, write down the agent's current completion claim. This is the "claimed" field in the gate report frontmatter and the "Claim recorded before gates ran" line in the body. It is a verbatim assertion — not a summary, not a forecast of what the gates will show.

This step is non-negotiable. A claim written after the gates ran measures nothing; the claimed-vs-verified delta is the perception-gap dataset for steward.

---

## Executing verification items

Execute each item in the spec's Verification section in order. Rules:

**Execute literally.** If an item says `run pytest tests/test_auth.py`, run that command. If it says "run `/gg:audit`" or "audit", invoke the `/gg:audit` gate command (see Standard implementations below). If it says "observe behavior X", observe behavior X. No implicit routing — the spec is the source of truth.

**Standard named implementations.** Two gate commands are pre-built and recognized by name in any spec's Verification section:

- `/gg:audit` — full parallel audit pass via `architect-reviewer` (coordinates security-auditor, code-auditor, doc-auditor, architect-reviewer, frontend-reviewer, ux-reviewer, product-reviewer). Returns PASS / FIX AND RE-AUDIT / NEEDS DISCUSSION. frontend-reviewer and ux-reviewer are skipped automatically when there are no UI changes. Spawning as a subagent is the correct invocation: the audit findings become proof for this gate item.
- `/gg:acceptance` — post-implementation product acceptance via `product-reviewer` + implementation walkthrough. Returns SHIP / FIX AND RE-CHECK / HOLD. Run this gate last (after all code gates pass) — the user experience check is meaningless if the code is known broken.

Route verdicts into the gate report:
- FIX AND RE-AUDIT / FIX AND RE-CHECK → do NOT fix mid-run; finish the full run, then route each finding to its plan item; re-run only the failed item after fixing.
- NEEDS DISCUSSION / HOLD → surface in the route section; do not count as PASS; do not block other items.

**Record proof per item.** The Proof column is the minimum useful excerpt: the specific failure message, the assertion that passed, the relevant counter value. Not full output, not stack traces, not session content. Enough for a reviewer to verify the result without re-running.

**BLOCKED is a valid result.** If an open escalation covers this item, mark it `BLOCKED (escalation NNN)`. Do not skip it; do not guess at the outcome.

**Do not edit code mid-gate run.** If a gate fails, finish the run, route the failure, then fix. Fixing mid-run contaminates the claimed-vs-verified delta.

## Building the gate report

File: `specs/<NN>-<slug>/gates/<YYYY-MM-DD>.md` (create the `gates/` directory if it doesn't exist).

Follow `templates/gate-report.md` exactly.

**Frontmatter:**
- `claimed: done | partial` — record this BEFORE the run, based on the agent's current belief
- `verdict: pass | fail` — fill after all items are run
- `attestation: agent` — this run was generated during a session, not by CI

**Verification items table.** One row per item from the spec's Verification section, in the same order. Verbatim item text from the spec.

**Claimed vs verified.** One or two sentences: what was claimed, what the gates found, where the gap was. If the agent claimed "done" and gates passed, say so. If the agent claimed "done" and a gate failed, name the gap explicitly — what was asserted but never tested.

**Traceability section.** Map every changed file to its plan item and requirement:

1. Get the list of changed files: `git diff --name-only main..HEAD` (or the appropriate base branch)
2. For each changed file, find the plan item(s) from `plan.md` that cover it and the requirement(s) those items serve
3. Files with no plan item are **orphans** — list them explicitly. Do not leave the orphan list empty by implication; state "No orphans" if that's the case.

For each orphan, route it:
- Explain how it came to exist (debugging, investigation, refactor spillover)
- Propose: plan revision (map it to an existing requirement), revert, or escalation

**Route section.** For each failure: which plan item reopens, or does it become a new escalation? For each orphan: plan revision proposed or revert. On full pass: "ready for PR; attach this report."

---

## Redaction rules

The gate report is safe to attach to a PR and may be read by people who were not in the session. Apply these rules to every field:

- `claimed:` field: a verbatim assertion only — no surrounding conversation context, no session quotes
- Proof column: minimum useful excerpt — the assertion, the counter, the specific failure message. No stack traces from internal tooling; no paths that expose personal directories; no API keys or credentials that appeared in output
- Nothing in any field is drawn from conversation history or session transcript

These rules are not optional. A report that exposes transcript content violates the redaction boundary. When in doubt, omit and note "proof available on request."

## After the gate run

**On full pass (all items PASS or N/A):**
- Set `verdict: pass`
- Route section: "ready for PR; attach this report as `specs/<NN>-<slug>/gates/<date>.md`"
- Proceed to create the PR with the report attached or referenced

**On any FAIL:**
- Set `verdict: fail`
- Do NOT create the PR
- Route each failure to its plan item; reopen it
- If the failure reveals a requirements gap (the gate is wrong, not the implementation), escalate — do not silently update the spec

**On any BLOCKED:**
- Set `verdict: fail` if any non-BLOCKED item also failed; otherwise set `verdict: blocked`
- Do NOT create the PR
- Route: resolve the blocking escalation first, then re-run the blocked items

**Re-runs:** After fixing failures, run only the failed items, not the full suite. Append a new gate report file with the current date; do not overwrite the original. The history of gate runs is intentional — steward reads the delta between claimed-vs-verified across runs.

## CI attestation

An agent-generated report (`attestation: agent`) is the author attesting their own work. It is better than no gates and better than a self-assessment, but it is still gameable.

A CI-attested report (`attestation: ci`) is generated by the Action during the agentic job — commands re-run independently, output captured, report written by the runner. This is the trustworthy form.

When CI is available: configure the Action to re-run the Verification items and generate the report. The agent's report becomes a draft; the CI report is the final attestation. The PR description should reference the CI report, not the agent report.

When CI is not available: use the agent report. Note its provenance. A reviewer who wants to verify can re-run the commands from the Proof column.

## Rules

- Claim before gates. Every time.
- Execute literally. No implicit routing.
- Proof per item. Minimum useful excerpt; nothing from session content.
- Open escalations block the gate run. Resolve first.
- Orphan changes are listed explicitly, not implied by their absence.
- Do not edit code mid-run.
- Do not create the PR until verdict is pass.
- Agent self-assessment is never a completion signal — including the gate run itself. The gates are the signal.
